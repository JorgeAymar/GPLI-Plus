# GLPI-inspirado ITSM/Asset Management — Next.js 16 + TypeScript + PostgreSQL

## Contexto

El usuario pidió "descargar GLPI y convertirlo a Next.js/Postgres", con alcance de **paridad de features completa**, para un negocio de **venta de servicios de implementación** (no venta del software en sí) sobre este producto.

GLPI es GPL-3.0, ~905K líneas de PHP, 442 tablas, 625 clases. Traducir su código/esquema literalmente crearía una obra derivada bajo GPL-3.0 — y "vender implementación" normalmente implica desplegar el software en/para infraestructura de cada cliente, lo cual sí cuenta como distribución bajo GPL-3.0 (a diferencia de un SaaS puro donde el proveedor controla el hosting). Para eliminar ese riesgo por completo, este plan es una **reimplementación independiente**: mismo dominio funcional que GLPI (ITSM + gestión de activos IT), pero con arquitectura, esquema de base de datos y código propios, diseñados desde cero. El repo de GLPI ya está clonado en `source/` solo como referencia de qué módulos/funcionalidades existen (catálogo funcional abajo) — ningún código ni DDL de GLPI se copia ni se traduce.

Dado el tamaño real del dominio (6 módulos, ~40 tipos de entidad, motor ITIL, RBAC multi-entidad), esto es un proyecto de meses construido por fases. Este plan cubre la arquitectura completa + roadmap de 7 fases, con **Fase 0 y Fase 1 detalladas a nivel de archivo/tabla** para arrancar de inmediato.

**Despliegue objetivo**: el usuario confirmó que el producto se instala en una máquina del cliente vía Docker (no SaaS centralizado) — esto refuerza directamente la necesidad del enfoque clean-room, ya que instalar/entregar el software en infraestructura del cliente es exactamente el escenario de "distribución" que activa las obligaciones copyleft de GPL-3.0 si hubiera código de GLPI de por medio. Por eso Fase 0 incluye, además del `docker-compose.yml` de desarrollo, un `Dockerfile` de producción para `apps/web` (y `apps/worker` desde Fase 3) pensado para instalación on-prem por cliente.

Después de aprobar este plan, el primer paso será invocar la skill **graphify** para documentar esta arquitectura como knowledge graph (pedido explícito del usuario), y luego arrancar el scaffold de Fase 0.

### Fase 6h — Plugins (cierre, sin código)

Decisión tomada explícitamente con el usuario (ver también las otras 2 decisiones de arquitectura documentadas al inicio de esta sección): **no se construye un sistema de carga dinámica de plugins de terceros**. GLPI resuelve esto con autoload de código PHP + funciones globales por convención de nombre (`plugin_<key>_*`) + hooks — replicar ese modelo en Node/Next.js significaría ejecutar código arbitrario subido por terceros en el mismo proceso del servidor (superficie de Remote Code Execution), un perfil de riesgo completamente distinto al resto de este proyecto.

**La extensibilidad del producto es Webhooks (6f) + API REST (6e)**, ambas ya construidas en esta misma fase — cubren el caso de uso real de integración (notificar a un sistema externo, leer/escribir datos vía API) sin necesitar ejecutar código de terceros. Si en el futuro se necesitara más que eso, la opción intermedia (descartada por ahora, dejada documentada) sería un sistema de **módulos propios de primera parte** — código TypeScript escrito por el propio equipo, registrado por configuración en el deploy, sin marketplace ni subida de código en caliente — nunca carga dinámica de terceros.

Con esto, **Fase 2 (Asset Management) está completa**: los 6 tipos de activo core están sembrados y operativos (2 con tabla de extensión + componentes, 4 genéricos vía `custom_fields`), más el framework completo para tipos custom sin migraciones, más gestión de software/licencias/instalaciones.

## Tercera ronda: fix de un bug de producción real (redirect loop) + feature nueva (tokens MCP personales)

### Bug real encontrado y corregido: loop infinito de redirects con sesión huérfana

Un usuario con una cookie de sesión JWT sintácticamente válida pero cuyo `userId` ya no existe en la base (ej. usuario borrado/desactivado, o volumen de Postgres recreado en dev) quedaba atrapado en un loop infinito: `/` → `/dashboard` (sin contexto, redirige a `/login`) → `/login` (el proxy creía que SÍ estaba logueado porque el JWT tenía `userId`, así que lo rebotaba a `/`) → repite para siempre. Reproducido de forma determinística con `curl --max-redirs` antes de arreglar, no solo teorizado.

Dos capas de la misma causa raíz (`apps/web/proxy.ts` y `apps/web/lib/session.ts` confiaban en la sola presencia de `userId` en el JWT, no en si ese usuario todavía existe):
- `lib/session.ts`: `requireAuthContext()` ahora hace `redirect("/login")` en vez de `throw new Error(...)` cuando `resolveAuthContext` devuelve `null` (sesión sintácticamente válida pero huérfana).
- `proxy.ts`: en vez de `!!req.auth` o solo chequear `userId` presente, ahora valida contra la DB con un helper liviano nuevo, `isActiveUserId()` (`packages/core/src/users/user-service.ts` — una sola query indexada por PK, no el `resolveAuthContext` completo de 5 queries que ya corre la capa de página, porque el proxy corre en *cada* request incluyendo cada POST de Server Action).

Verificado extremo a extremo con una sesión huérfana real (login real → desactivar el usuario en DB → repetir el request): antes, loop infinito confirmado con `curl --max-redirs 10` agotando el límite; después, un solo redirect limpio a `/login`.

### Feature nueva: tokens de acceso personales + servidor MCP + preferencia de idioma

Motivado por: "el usuario quiere conectar la app a Claude/clientes MCP con su propio token, gestionado desde una página de cuenta personal". Diseñado con el flujo de brainstorming (spec en [`docs/superpowers/specs/2026-07-18-mcp-server-personal-tokens-design.md`](superpowers/specs/2026-07-18-mcp-server-personal-tokens-design.md), plan de implementación de 10 tareas en [`docs/superpowers/plans/2026-07-18-mcp-server-personal-tokens.md`](superpowers/plans/2026-07-18-mcp-server-personal-tokens.md)), ejecutado con subagentes (implementador + revisor de spec + revisor de calidad, por tarea).

**Modelo de datos**: `api_clients` (ya existía, tokens bearer estilo Stripe a nivel de *entidad* para `/api/v1`) se extendió con una columna `user_id` nullable + constraint `CHECK` (`api_clients_entity_xor_user`) que fuerza que **exactamente uno** de `entity_id`/`user_id` esté seteado — nunca ambos, nunca ninguno, garantizado a nivel de base de datos, no solo en el código de la app. Un token con `user_id` seteado es "personal" (prefijo `pat_`, dueño = un usuario); uno con `entity_id` sigue siendo el modelo de siempre (prefijo `sk_`, dueño = una entidad, gestionado por un admin).

**Autorización de tokens personales**: nada de una lista de "scopes" elegida a mano al crear el token (que es como funcionan los tokens de entidad). Cada llamada de un token personal resuelve el `AuthContext` real del usuario dueño (`resolveAuthContext`) y aplica el mismo `requireRight()` que ya usa toda la UI — un token personal nunca puede leer más de lo que ese usuario podría ver logueado normalmente.

**`/account` ("Mi cuenta")**: página nueva, deliberadamente en la ruta `/account` y no `/profile` porque `/administration/profiles` ya significa "roles RBAC" en esta app — usar el mismo nombre para "mi cuenta personal" hubiera sido confuso. Desde ahí: crear/listar/revocar tokens personales (la key cruda se muestra una sola vez, igual que en `/setup/api-clients`), y un selector de idioma preferido (`es`/`en`/`pt`/`fr`/`it`/`de`) que **solo guarda la preferencia** — la columna `users.language` ya existía en el schema desde una fase anterior pero nunca se había conectado a nada; conectarla no implica que la app ya traduzca su UI (eso es un motor de i18n completo, deliberadamente fuera de alcance, documentado como próximo proyecto separado).

**`/api/mcp`**: servidor MCP (Model Context Protocol) real usando el SDK oficial `@modelcontextprotocol/sdk` (`McpServer` + `WebStandardStreamableHTTPServerTransport`, transporte stateless — cada tool call resuelve el contexto de auth de cero, no hay estado compartido entre requests, ni tiene sentido cachear el `McpServer` entre requests porque el SDK explícitamente prohíbe reusar un transporte stateless). Las tools (`list_<itemtype>`/`get_<itemtype>`, 10 en total: 5 tipos × 2) se generan dinámicamente iterando `ITEMTYPE_REGISTRY` (ya existía, el mismo registro que respalda `/api/v1`) — agregar un itemtype nuevo al registro lo hace aparecer automáticamente como tool MCP, sin tocar la ruta. Un token de entidad es rechazado explícitamente con 401 si se usa contra `/api/mcp` (los dos tipos de token no son intercambiables), y — chequeo simétrico agregado en la misma pasada — un token personal ahora también es rechazado con 401 si se usa contra `/api/v1` (antes de este fix, el cambio de `entityId` a nullable rompía el typecheck de esa ruta; arreglarlo cerró de paso el hueco de que un token personal pudiera colarse en la API de entidad).

**Hallazgo de seguridad real, mitigado pero no cerrado del todo (deliberado, documentado explícitamente en el código)**: `get_<key>` (y el `/api/v1/[itemtype]/[id]` preexistente del que hereda el patrón) no verifica que el registro pertenezca a la entidad activa del que llama — cualquier id válido de cualquier entidad se devuelve si existe. Bajo REST esto requiere que un humano arme deliberadamente un UUID ajeno; bajo MCP un agente LLM va a seguir naturalmente IDs cruzados que aparecen en los datos (tickets vinculados, cambios relacionados) como comportamiento normal de "dame más contexto", con o sin intención adversarial — la exposición real es mayor acá que en su origen. Cerrarlo bien implica cambiar la firma de 5 funciones getter + todos sus ~15 call sites (incluyendo páginas de detalle humanas que tienen el mismo hueco hoy, ej. `/assistance/tickets/[id]`) — se dejó fuera de esta feature a propósito (necesita su propio diseño) y en su lugar se agregó una advertencia explícita en la descripción de la tool MCP (que un agente que la llama puede leer) más un comentario extenso en el código listando el blast radius exacto para cuando se aborde.

**Verificación real, no solo tests**: además de 690 tests unitarios y 152 E2E (detalle abajo), se probó el protocolo MCP de punta a punta con un cliente MCP real (el mismo SDK que usarían Claude Desktop/Claude Code) contra el servidor corriendo — no solo códigos de estado HTTP vía curl, sino un handshake MCP real, `listTools()` devolviendo las 10 tools esperadas, y `callTool()` devolviendo JSON parseable. Ese test (`apps/web/app/api/mcp/mcp-route.integration.test.ts`) quedó deliberadamente separado del `pnpm test` por defecto (`apps/web/vitest.config.ts` lo excluye, `apps/web/vitest.integration.config.ts` + el script `test:mcp-integration` lo corren aparte) porque requiere un `npm run dev` corriendo — de lo contrario `pnpm test`/CI se vuelven no determinísticos según si hay un server externo levantado.

**Reinicios de dev server durante la sesión (documentado porque no es un bug de código)**: el dev server se cayó varias veces durante esta sesión por motivos ambientales, no por el código escrito — una vez por `node_modules` desincronizado tras instalar la dependencia del SDK sin reiniciar, dos veces por memoria del sistema agotada (confirmado con `vm_stat`: ~64MB libres en toda la máquina, con 4 sesiones de Claude Code + VS Code + Chrome + un antivirus al 100%+ CPU corriendo en simultáneo — no relacionado a este repo). Cada caída se diagnosticó leyendo el log real antes de reiniciar, nunca se asumió "reiniciar y listo" sin confirmar la causa.

Commits relevantes (orden cronológico, `git log` tiene el detalle completo): fix del redirect loop, dependencia del SDK, schema+migración de `api_clients`, servicios core de tokens personales + idioma, server actions + UI de `/account`, endpoint `/api/mcp`, test de integración MCP real, aislamiento de ese test del `pnpm test` por defecto, e2e de `/account` (con hardening post-review: `try/finally` en el test de idioma, regex exacto de la key, navegación fresca antes de revocar).

## Acciones inmediatas (antes de tocar código de Fase 2)

El agente que escribió el Dockerfile/CI corrió `docker build` real y encontró **2 errores de TypeScript preexistentes** que bloquean `pnpm build`/`pnpm typecheck`/CI. Diagnosticados con `tsc --traceResolution`, causa raíz confirmada:

1. **`packages/db/src/schema/auth.ts`**: `import type { AdapterAccountType } from "next-auth/adapters"` falla (`TS2307`) porque `packages/db` no declara `next-auth` como dependencia — y no debería (es un paquete de schema agnóstico de framework). pnpm con node_modules estricto no deja ver ese paquete desde ahí. **Fix**: borrar el import y quitar `.$type<AdapterAccountType>()` de la columna `type` de `accounts` (queda como `text("type").notNull()` simple).
2. **`apps/web/lib/auth.ts`**: `declare module "next-auth/jwt" { interface JWT {...} }` falla (`TS2664`) porque `next-auth/jwt.d.ts` es un barrel puro (`export * from "@auth/core/jwt"`) — TypeScript no puede aumentar un módulo que solo reexporta, es una limitación documentada de Auth.js v5. **Fix**: cambiar el `declare module` a `"@auth/core/jwt"` directamente, y agregar `@auth/core` (versión `0.41.2`, ya resuelta transitivamente vía `next-auth`) como devDependency explícita en `apps/web/package.json` para que sea resoluble desde ahí.
3. Verificar con `pnpm install && pnpm --filter @itsm/web typecheck` (0 errores) antes de empezar Fase 2.

## Stack técnico (decisiones)

| Decisión | Elección | Por qué |
|---|---|---|
| ORM | **Drizzle ORM** + drizzle-kit | Query builder SQL-first, soporte nativo de `ltree`/JSONB/CTEs recursivas que necesitamos en casi toda tabla; sin motor de runtime pesado como Prisma. |
| Auth | **Auth.js v5**, sesión **JWT**, adapter Drizzle | Credentials provider (nuestro login usuario/password) solo soporta JWT en Auth.js — no hay fila OAuth de la cual colgar una sesión en DB. "Cambiar entidad/perfil activo sin re-login" se resuelve con el trigger `update()` sobre el JWT. Credentials provider hoy, LDAP/OIDC/SAML en Fase 6 (planificar sobre JWT). |
| UI | **Tailwind + shadcn/ui** | Componentes accesibles, código propio en el repo (sin lock-in), soporta dos shells (Central vs Simplified). |
| Data layer | **Server Components + Server Actions** sobre una capa de servicio compartida (`packages/core`), también expuesta vía **route handlers REST** (`/api/v1/...`) | Lógica de negocio se escribe una vez y se expone dos veces (UI interna + API pública), sin duplicar con tRPC. |
| Jobs/cron | **pg-boss** (cola sobre Postgres) en `apps/worker` | Encolar jobs (SLA, notificaciones, recurrencias) en la misma transacción que el escritor; sin infra Redis extra. |
| Storage | Adapter (`LocalFsAdapter` dev / `S3Adapter` prod) + tabla `documents` + `document_items` polimórfica | Un solo mecanismo de adjuntos reusado por todos los módulos. |
| Testing | Vitest (unit + integración con Postgres dockerizado, transacciones con rollback) + Playwright (E2E flujos críticos) | Cobertura rápida de lógica + verificación real de flujos multi-entidad/permisos/tickets. |

**Monorepo**: pnpm workspaces + Turborepo — `apps/web` (Next.js 16), `apps/worker` (desde Fase 3), `packages/db` (schema Drizzle + migraciones), `packages/core` (servicios, Zod, permisos, adapters).

**Notas Next.js 16**: cuidado con `"use cache"` (Cache Components) en cualquier función entity-scoped — nunca cachear sin incluir entidad/usuario en la key (riesgo de fuga de datos entre tenants). `proxy.ts` (antes `middleware.ts`) corre en runtime Node.js siempre (ya no es configurable — lanza error si se especifica `runtime` en su `config`), así que puede resolver `auth()` completo por request.

## Esquema PostgreSQL (conceptual, propio — no copiado de GLPI)

- **Entities**: `entities(id, parent_id, name, path ltree, level, is_active, ...)` — `ltree` + GIST index para queries de subárbol O(index). Requiere `CREATE EXTENSION ltree`.
- **Users/Auth**: `users`, `accounts`/`sessions`/`verification_tokens` (shape Auth.js estándar, sin columnas extra — `activeEntityId`/`activeProfileId` viven en el JWT, no en la tabla `sessions`, ver "Estado actual" arriba); `groups` (árbol), `user_groups`.
- **RBAC**: `profiles(id, name, interface enum[central,simplified], ...)`, `profile_module_rights(profile_id, module_key, rights bitmask)`, `user_profiles(user_id, profile_id, entity_id, is_recursive, is_default)` — permite N perfiles × N entidades por usuario.
- **Assets (framework extensible, sin migraciones para tipos custom)**: `asset_definitions` (tipos, incl. custom admin-creados) + `asset_field_definitions` (campos custom por tipo) + tabla polimórfica única `assets` (columnas comunes + `custom_fields JSONB` validado dinámicamente con Zod) + **tablas de extensión finas** solo para tipos core con integridad relacional real (`computers`, `network_equipment`, `asset_components` para CPU/RAM, etc.). Software/License con tablas propias (`software`, `software_versions`, `software_licenses`, `asset_software_installations`) por sus necesidades relacionales de licenciamiento.
- **ITIL (Ticket/Problem/Change)**: tablas separadas tipadas (`tickets`, `problems`, `changes`) compartiendo un set de columnas común (enum `itil_status`) vía interfaz TS + Zod compartidos (no herencia de tablas Postgres, frágil con FKs), más **satélites polimórficos** compartidos: `itil_actors`, `itil_timeline_items`, `itil_validations`, `itil_costs`, `itil_sla_assignments` vía `(itil_type, itil_id)`. `sla_policies`/`sla_escalation_levels` (Fase 3).
- **Cross-cutting**: `dropdown_categories`/`dropdown_items` (sistema genérico reusado por FKs de assets y custom fields tipo dropdown), `documents`/`document_items` (adjuntos polimórficos), `notification_templates`/`notification_events`/`queued_notifications`, `audit_log` (genérico, indexado por `object_type, object_id`).

### Esquema DB nuevo (7 archivos en `packages/db/src/schema/`)

- `dropdowns.ts`: `dropdown_categories(id, key unique, name, is_system, ...)`, `dropdown_items(id, category_id, entity_id, parent_id self-fk, name, comment, is_active, ...)`. **Validación dinámica de `custom_fields`** — nuevo `packages/core/src/assets/dynamic-schema.ts`:
```ts
function zodForField(field: AssetFieldDefinition): z.ZodTypeAny { /* mapea field_type -> z.string()/z.coerce.number()/etc, .nullable().optional() si !isRequired */ }
export function buildDynamicSchema(fieldDefs: AssetFieldDefinition[]): z.ZodObject<...> // .strict() — rechaza keys desconocidas
export async function validateCustomFields(assetDefinitionId: string, customFields: unknown): Promise<Record<string, unknown>>
```
`createAsset()`/`updateAsset()` llaman `validateCustomFields()` antes de escribir — así `custom_fields` nunca se persiste sin pasar el schema del admin para ese tipo, y `.strict()` evita que campos borrados dejen basura aceptada en escrituras futuras.
